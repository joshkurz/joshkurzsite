## Terraform Findings

### MEDIUM — IAM over-permission on Lambda role
`terraform/dynamodb.tf:219-231`

The ratings-aggregator Lambda needs only `GetItem` + `UpdateItem` on the stats table and `GetRecords`/`GetShardIterator` on the stream. Instead the policy grants:

```hcl
"dynamodb:GetItem",
"dynamodb:PutItem",
"dynamodb:UpdateItem",
"dynamodb:DeleteItem",   # not needed
"dynamodb:Query",        # not needed
"dynamodb:Scan"          # not needed — and dangerous
```

`DeleteItem` and `Scan` on both tables is over-permissioned and violates least privilege. A compromised Lambda could delete or dump all ratings/stats data.

**Fix:** Scope the Lambda policy to only what it uses:
```hcl
Action = [
  "dynamodb:GetItem",
  "dynamodb:UpdateItem"
]
Resource = [
  aws_dynamodb_table.joke_stats.arn,
  "${aws_dynamodb_table.joke_stats.arn}/index/*"
]
```

Keep the separate stream-access statement as-is.

### LOW — `.terraform.lock.hcl` is gitignored (should be committed)
`.gitignore:48`

```
terraform/.terraform.lock.hcl
```

The lock file pins provider checksums for reproducible, tamper-evident builds. Gitignoring it means every `terraform init` resolves providers fresh — a supply chain risk (provider version drift or substitution). This file contains no secrets.

**Fix:** Remove `.terraform.lock.hcl` from `.gitignore` and commit it.

---

### LOW — Local Terraform state, no remote backend
`terraform/main.tf:18-25`

The S3 remote backend is commented out. Local state means:
- State is not shared across teammates/CI
- State file (which contains resource IDs and ARNs) is not version-controlled or encrypted
- Concurrent applies can corrupt state

**Fix:** Uncomment and configure the S3 backend with a dedicated locking DynamoDB table and `encrypt = true`.

---

### LOW — DynamoDB Stream view type sends more data than needed
`terraform/dynamodb.tf:77-78`

```hcl
stream_view_type = "NEW_AND_OLD_IMAGES"
```

The Lambda aggregator only reads `NewImage` from stream records (see `lambda/ratings-aggregator/index.js:56`). `NEW_AND_OLD_IMAGES` doubles the data volume in the stream and exposes previous ratings values unnecessarily.

**Fix:**
```hcl
stream_view_type = "NEW_IMAGE"
```

---

### INFO — Default `environment` is `prod`
`terraform/variables.tf:7`

```hcl
default = "prod"
```

A `terraform apply` with no arguments targets production. Defaulting to `dev` or requiring explicit `-var="environment=prod"` reduces the blast radius of accidental applies.

---

### INFO — No customer-managed KMS encryption on DynamoDB tables
`terraform/dynamodb.tf:9, 98`

DynamoDB encrypts at rest using AWS-owned keys by default. No `server_side_encryption` block is configured. For a public dad-jokes app this is acceptable, but for compliance or higher-sensitivity workloads a CMK would be appropriate.

### MEDIUM — Rate limits are too permissive (1000 req/min)
`middleware.js:5-9`

```js
const LIMITS = {
  '/api/ai-joke': 1000,
  '/api/speak': 1000,
  '/api/custom-jokes': 1000,
}
```

1000 requests per minute per IP is 16+ req/sec — effectively no protection for paid endpoints. The previous review had limits of 10 for ai-joke/speak and 5 for custom-jokes. These were raised by 100x somewhere.

**Attack scenario:** An attacker can still burn OpenAI quota at 16 req/sec per IP (thousands of dollars/hour if using multiple IPs).

**Fix:** Lower limits to match cost exposure:
```js
const LIMITS = {
  '/api/ai-joke': 10,     // ~$0.01/req at gpt-4o-mini rates
  '/api/speak': 10,       // TTS costs money per character
  '/api/custom-jokes': 20,
}
```

### MEDIUM — `author-jokes.js` still lacks input validation
`pages/api/author-jokes.js:20-24`

```js
const { author } = req.query
if (!author) {
  return res.status(400).json({ error: 'Author parameter is required' })
}
```

Flagged in the 2026-03-05 review — **still not fixed**. Two problems:
1. `req.query.author` can be a string array if the URL is `?author=x&author=y`. `typeof author` will be `'object'`, the null check passes, and `AUTHOR#x,y` is used as the GSI key.
2. No length cap — a 4000-character author string burns DynamoDB read capacity.

**Fix:**
```js
const { author } = req.query
if (!author || typeof author !== 'string' || author.length > 200) {
  return res.status(400).json({ error: 'Author parameter is required' })
}
```

## Security Wins (Unchanged)

- No hardcoded secrets — all credentials via `process.env`, `.gitignore` excludes `.env*` and `*.tfstate`
- DynamoDB parameterized queries throughout — no string-interpolated expressions
- Lambda rating validation — `lambda/ratings-aggregator/index.js:64` validates 1-5 integer range before any DynamoDB write
- AI content moderation — user jokes go through GPT moderation before storage
- Voice allowlist — `speak.js` validates voice against an explicit allowlist
- IAM least-privilege for app access — `app_dynamodb_access` policy scoped to only `PutItem`, `GetItem`, `Query`
- CloudWatch log retention set to 14 days (avoids unbounded log storage cost)
- DynamoDB PITR enabled by default (`enable_point_in_time_recovery = true`)
